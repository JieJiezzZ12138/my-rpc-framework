# RPC 框架开发笔记：从零到动态代理（V2.0）

## 0. 为什么要开发？
在分布式系统中，服务 A 想调服务 B 的方法，不能像本地调用那样直接 `new` 对象。
**RPC (Remote Procedure Call)** 的核心目标：**让跨网络的方法调用，在代码层面上看起来跟本地调用一模一样。**

为了实现这个目标，我把项目拆成了四个模块：
* `rpc-api`：契约（接口和实体类）。
* `rpc-core`：发动机（序列化、代理、网络传输）。
* `rpc-provider`：服务提供方。
* `rpc-consumer`：服务消费方。

---

## 1. 核心协议设计：RpcRequest
网络传输本质是二进制流，所以我必须定义一个**通用的通信报文**。

### 踩坑记录：为什么必须传 `paramTypes`？
最初我只想传方法名和参数值，但发现 Java 有**方法重载**。
* 比如 `sayHello(String)` 和 `sayHello(User)`。
* 如果只给个参数值，服务端反射查找方法时会因为找不到唯一匹配而报错。
* **结论**：必须把参数的 Class 类型数组也传过去，反射 `getMethod(name, paramTypes)` 才能精准定位。

---

## 2. 网络传输：基于 BIO 的 Socket 通信
目前 V2.0 阶段，我选了最基础的 **BIO (Blocking I/O)**。
* **服务端**：用 `ServerSocket.accept()` 阻塞等待连接。
* **客户端**：用 `Socket` 建立连接。
* **序列化**：暂时用 Java 原生的 `ObjectOutputStream`，简单粗暴，但实体类必须实现 `Serializable`。

**经验**：这里用了 **Try-with-resources** 语法。Socket 和流用完必须关，否则端口被占用，下次启动直接报错。自动关闭比手动写 `finally` 优雅得多。

---

## 3. 服务端逻辑：反射 (Reflection) 动态分发
服务端不写死调哪个类，而是通过 `RpcRequest` 传过来的信息动态去找。
1. 拿到接口名，在本地找到对应的实现类。
2. 拿着方法名和参数类型，通过反射拿到 `Method` 对象。
3. 执行 `method.invoke(target, args)` 并把结果写回 Socket。



---

## 4. V2.0 的灵魂：JDK 动态代理
这是项目从“手动档”变“自动档”的关键。

### V1.0 的痛点：
客户端得自己 new RpcRequest，自己调 RpcClient，业务代码里全是网络通信，太乱了。

### V2.0 的进化：
我写了一个 `RpcClientProxy` 类。
* 它在内存里给接口（比如 `HelloService`）生成一个**代理对象**。
* 客户端调用 `helloService.sayHello()` 时，其实是在调代理的 `invoke` 方法。
* **屏蔽细节**：在 `invoke` 内部自动完成封包、发请求、收结果。
* **结果**：客户端代码一行网络逻辑都没有，实现了**调用透明化**。

---

## 5. 复盘

### Q1：项目怎么体现“解耦”？
* 首先是模块化设计；其次是动态代理。Consumer 只需要依赖 `rpc-api` 的接口，不需要知道 Provider 的具体实现，也不需要知道底层的 Socket 是怎么发包的。

### Q2：目前的实现有什么明显的性能问题？
* **阻塞！** 服务端是单线程 `while(true)`。如果一个请求处理慢了，后面的所有请求都会排队挂起。这是典型的 **BIO 瓶颈**。

### Q3：为什么序列化 ID（serialVersionUID）很重要？
* 因为网络传输的是二进制。如果我在传输过程中改了实体类的代码（比如加个字段），而序列化 ID 没定死，反序列化就会报错。手动指定 `1L` 是为了保证版本兼容。

---

## 6. 下一步规划 (Roadmap)
V2.0 虽然能跑，但还是个“单线程弱鸡”。
1. **引入线程池**：让主线程只接客，干活的事交给线程池。
2. **研究 Netty**：把 BIO 换成 NIO，这才是工业级 RPC 的底座。
3. **自定义协议**：现在直接传整个 Object 太重了，以后要设计精简的字节协议。

# V3.0 线程池并发重构

## 1. 为什么要重构？（V2.0 的痛点）
V2.0 虽然跑通了代理流程，但服务端是个“单线程弱鸡”。
* **现象**：serverSocket.accept() 之后直接在主线程处理 I/O 和反射。
* **后果**：如果一个请求耗时 3 秒，这 3 秒内服务端无法响应任何其他客户端。这种 **同步阻塞 BIO** 模式完全无法商用。

## 2. 核心架构升级：接客与干活分离
为了解决阻塞，我把服务端改成了 **“多线程 BIO”** 模型。

### 2.1 任务封装：RequestHandlerThread
实现 Runnable 接口，只负责“读数据 -> 反射调用 -> 回写数据”，符合**单一职责原则**。

### 2.2 调度中心：ThreadPoolExecutor
显式创建自定义线程池，配置核心参数：核心线程 5，最大线程 10，有界队列 100（防止 OOM）。

## 3. 并发性验证（实战记录）
**测试方法**：HelloServiceImpl 加 Thread.sleep(3000)，客户端双线程同时发起请求。
**结果**：两个客户端均在约 3 秒收到响应。并发重构成功。

## 4. 深度复盘：线程池的局限
虽然有了线程池，但模型依然是 **“一个连接占一个线程”**。在海量长连接场景下依然会爆掉。V4.0 必须考虑引入 **Netty (NIO)**。

# V4.0 服务注册表与动态路由分发

## 1. 为什么要重构？（V3.0 的瓶颈）
V3.0 虽然解决了并发问题，但 `RpcServer` 内部硬编码了一个具体的服务对象。
* **痛点**：一个端口只能为一个接口服务。如果想同时提供 `HelloService` 和 `PingService`，难道要开两个端口？这显然不符合微服务“高内聚、低耦合”的逻辑。
* **本质**：通信引擎（Socket）与业务逻辑（Service）高度耦合。

## 2. 核心架构升级：引入“前台接待处” (ServiceProvider)
我引入了 **服务注册表 (Service Registry)** 机制，将服务端从“专卖店”升级为“百货商场”。

### 2.1 注册中心：ServiceProvider
* **数据结构**：采用 `ConcurrentHashMap<String, Object>`。
* **Key 的选择**：使用接口的 **全限定名 (Canonical Name)**。客户端手里只有接口契约，这是双方唯一的“对暗号”凭证。
* **线程安全**：由于工作线程会并发查表，必须使用 `ConcurrentHashMap` 的分段锁/CAS 机制，防止在高并发下出现数据丢失或死循环。


### 2.2 协议增强：RpcRequest 升级
为了配合路由，`RpcRequest` 新增了 `interfaceName` 字段。
* **逻辑**：客户端代理自动填充接口名 -> 发送到服务端 -> 服务端根据名字去 Map 里“搜”出对应的实现类实例。

## 3. 并发路由实战验证（硬核反馈）
**测试场景**：服务端同时注册 `HelloService` (耗时 3s) 和 `PingService` (即时返回)。
**实测日志**：
* 【线程 1】发起 `sayHello` 请求...
* 【线程 2】发起 `ping` 请求...
* **【线程 2】返回结果: Pong! | 实际耗时: 27ms**
* **【线程 1】返回结果: RPC 调用成功 | 实际耗时: 3034ms**

**结论**：
1.  **路由准确**：两个请求进了同一个 9000 端口，但精准投递到了不同的实现类。
2.  **真并发**：耗时长的 `sayHello` 并没有阻塞即时返回的 `ping`。

## 4. 深度复盘：目前的“技术肌肉”
* **解耦程度**：现在 `RpcServer` 变成了一个通用的容器，只要往 `ServiceProvider` 里注册，它就能跑。
* **反射技巧**：通过 `method.getDeclaringClass().getCanonicalName()` 在代理层自动获取接口名，实现了调用的完全透明化。

## 5. 下一步规划 (Roadmap)
虽然路由和并发都通了，但底层依然是 **BIO (Blocking I/O)**。
* **隐患**：每个连接依然 1:1 占用线程。若有大量“僵尸连接”占着坑不发消息，线程池会迅速枯竭。
* **目标**：**V5.0 引入 Netty**。进行“心脏移植”，从 BIO 进化到 NIO，实现单机万级的吞吐量。

# V5.0 高性能通信重构：引入 Netty (NIO)

## 1. 为什么要重构？（V4.0 的隐患）
在 V4.0 之前，我们底层的网络通信一直使用的是 Java 原生的 `Socket`。
* **致命痛点 (BIO 的局限)**：原生 Socket 是**同步阻塞 I/O (BIO)**。这意味着服务端为每一个连上来的客户端都必须分配一个专属线程。
* **C10K 问题**：如果同时有 10000 个客户端连接（即使它们什么数据都不发，只是保持连接），服务端的线程池也会瞬间耗尽资源，导致系统崩溃。

## 2. 核心架构升级：心脏移植与依赖倒置
为了解决并发上限问题，V5.0 引入了工业级的高性能网络框架 **Netty**。

### 2.1 依赖倒置原则 (DIP) 的工程实践
为了不让系统与某个特定的网络框架（如 Socket 或 Netty）死死绑定，我们在 `rpc-core` 中进行了**接口抽象**：
* 提取出 `RpcServer` 和 `RpcClient` 接口。
* 动态代理层 (`RpcClientProxy`) 和启动类 (`ProviderMain`) **只依赖接口，不依赖具体实现**。
* **收益**：网络层的具体实现被彻底解耦，未来如果想从 Netty 切换到别的通信框架，业务层和代理层可以做到**零代码修改**。

### 2.2 引入 Reactor 线程模型 (NIO)
* **BossGroup & WorkerGroup**：服务端不再是“一连接一线程”，而是基于事件驱动。
* `bossGroup` 专门负责处理 TCP 连接请求；`workerGroup` 专门负责处理真正的 I/O 读写。
* **非阻塞**：当客户端没有发送数据时，Worker 线程不会阻塞等待，而是去处理其他活跃的连接，极大地压榨了 CPU 性能，实现了真正的**高吞吐量**。

## 3. 并发与非阻塞实战验证
在替换为 Netty 底座后，我们重新执行了 V4.0 的并发测试（3秒耗时服务 vs 即时响应服务）。
**实测日志反馈**：
* 【线程 A】开始请求 HelloService...
* 【线程 B】开始请求 PingService...
* **【线程 B】返回结果: Pong! 来自 V5.0 Netty 容器的响应 | 实际耗时: 79ms**
* **【线程 A】返回结果: RPC 调用成功 | 实际耗时: 3183ms**

**结论**：在底层改为 Netty 且复用同一个 9000 端口的情况下，系统依然完美保持了多服务路由的精确性，以及线程的非阻塞并发能力。

## 4. 深度复盘：当前架构的“神”与“形”
* **契约层**：`rpc-api` 定义通用接口。
* **代理层**：`RpcClientProxy` 透明化封装请求包。
* **网络层**：Netty 提供极高吞吐量的传输通道。
* **路由层**：`ServiceProvider` 基于 `ConcurrentHashMap` 实现动态寻址。
* **执行层**：通过反射 `Method.invoke()` 动态执行本地业务逻辑。

## 5. 下一步规划 (Roadmap): 走向分布式
目前的系统已经是一台性能极高的“单机服务器”，但由于 IP 地址依然是硬编码的，它还不具备“分布式”的灵魂。
* **V6.0 目标：引入 Zookeeper 分布式注册中心**。
* **愿景**：让服务提供者自动注册 IP 到云端，客户端自动从云端发现服务，彻底抛弃本地写死的 `127.0.0.1`。

#  V6.0：别再 Hardcode IP 了！引入 ZK 实现“全自动导航”

## 1. 痛点：V5.0 还是“手动挡”
虽然 Netty 很猛，但 `127.0.0.1:9000` 是写死的。
* **尴尬场景**：服务端换个端口，客户端就得改代码重启；要是开了 5 台服务器，客户端根本不知道去哪。
* **目标**：实现“服务自发现”。客户端只管调接口，地址让 ZK 自动带路。

---

## 2. 骚操作：Zookeeper 是怎么“带路”的？
我把 Zookeeper 当成一个**“实名制动态通讯录”**：
1.  **服务端（Provider）**：启动时去 ZK 报到，在 `/my-rpc/接口名` 下面插个旗子，写上自己的 IP。
2.  **客户端（Consumer）**：要调接口前，先去 ZK 搜一下这个接口名下面有哪些旗子，拿走 IP 再连。



---

## 3. 核心黑科技：临时节点 (Ephemeral)
这是全篇最硬核的地方。我在注册时选了 **临时节点**：
* **原理**：只要服务端活着，ZK 就留着它的 IP；服务端一旦宕机（或者我强行关掉），ZK 发现心跳断了，**秒级自动删掉**那个 IP 节点。
* **爽点**：这就是“高可用”的雏形！客户端永远不会拿到一个“死地址”。

---

## 4. 架构大改动 (解耦才是真香)
为了不让 Netty 跟 ZK 死死捆绑，我玩了一手 **接口抽象**：
* 定义了 `ServiceRegistry` 和 `ServiceDiscovery` 接口。
* **服务端**：启动后自动遍历本地 Map，把所有服务名一次性推送到 ZK。
* **客户端**：彻底删掉 IP 端口参数，发请求前自动去 ZK “问路”。
* **工具人**：用了 **Curator**。这玩意儿比原生 ZK API 好用一万倍，自带重试策略，网络抖动也不怕。

---

## 5. 验收：分布式初体验
* **Provider 启动**：控制台刷出 `【ZK 注册】服务已挂载`，去 ZK 命令行输入 `ls` 真的能看到节点！
* **Consumer 启动**：代码里啥 IP 没写，直接调用成功。
* **结论**：这才是真正的分布式 RPC 框架，从此 IP 是路人。

---

## V6.5 版本更新日志：集群负载均衡与动态配置

**发布时间**：2026-04-08
**核心目标**：解决单点调用瓶颈，实现客户端请求的高可用分发与框架配置的热插拔。

### 新增特性 (Features)
1. **负载均衡引擎 (Load Balance)**：
   * 抽象 `LoadBalance` 顶级接口，规范路由算法标准。
   * 实现 `RandomLoadBalance` (随机算法)：适用于大规模集群的概率性均摊。
   * 实现 `RoundRobinLoadBalance` (轮询算法)：基于 `AtomicInteger` 实现线程安全的绝对均匀分发。
2. **动态配置中心 (Configuration)**：
   * 新增 `PropertiesUtil` 工具类，支持在框架启动时自动加载 `classpath:rpc.properties`。
   * 彻底告别硬编码，支持通过修改 `rpc.loadbalance.strategy` 属性无缝切换底层的路由算法。

### 架构优化 (Refactor & Security)
1. **服务发现改造**：重构 `ZkServiceDiscovery`，彻底剥离写死的单节点路由逻辑，将节点选择权交由负载均衡器代理。
2. **防御性编程 (Defensive Programming)**：在客户端 `NettyRpcClient` 中增加空节点拦截逻辑。当注册中心无存活的 Provider 时，优雅地抛出路由异常，避免底层触发惨烈的 `NullPointerException`。
